# BFC QR Ordering – Cursor Rules

## File Creation Policy (STRICT)

DO NOT create new markdown files.

If a prompt would normally generate:
- SHOTS_PRICING_MODES.md
- PER_SIZE_SHOT_DEFAULTS.md
- IMPLEMENTATION_SUMMARY.md
- FEATURE_UPDATE.md
- FIX_NOTES.md
- ANY root-level .md file

Instead:
- Update the correct existing file inside /docs
- Merge content into pos-register-spec.md if POS-related
- Merge schema changes into architecture/database-models.md
- Merge enum changes into architecture/enums-and-constants.md

Creating a new markdown file is a violation of project rules.

## Documentation Discipline

Do not create new root-level .md files.

When documenting a change:
- Update the appropriate file inside /docs.
- Do not create implementation summary files.
- Merge new feature documentation directly into the relevant spec file.

Feature categories:

- Schema change → architecture/database-models.md
- New enum → architecture/enums-and-constants.md
- POS behavior change → pos/pos-register-spec.md
- UI layout change → ui/pos-register-ui.md
- QR behavior → qr/qr-order-flow.md
- Staff logic → staff/staff-system.md

## Documentation Source of Truth

- Always read `/docs/pos/pos-register-spec.md` before modifying POS logic.
- If documentation conflicts, `pos-register-spec.md` wins.
- Do not rely on legacy markdown files.

## Architecture Constraints

- Local-first only.
- Single store: `store_1`.
- SQLite only.
- Fastify + Prisma backend.
- Next.js frontend.
- No cloud. No sync layer.

## Payment Rules

- Do NOT mix QR and POS payment systems.
- QR supports only:
  - PAYMONGO (Pay Online)
  - CASH (displayed as "Pay at the Counter")
- The enum remains `CASH`. Do not rename it.
- POS payment methods are configurable via StoreConfig.
- Never reintroduce `GCASH_MANUAL`.

## Staff Rules

- Payment actions require active staff.
- Cart browsing must remain staff-free.
- Staff login is embedded in cart header (UTAK-style).

## Split Payment

- Split button must not create a sale on click.
- Sale is created only on Charge.
- Enforce Remaining == 0 before charge.

## Modification Discipline

When changing behavior:
1. Update `pos-register-spec.md` first.
2. Then update code.
3. Keep changes localized.
4. Do not introduce new libraries.
5. Do not refactor unrelated areas.

If unsure where documentation belongs, ask instead of creating a file.

## Debug / Fix Documentation Policy

Do NOT create standalone .md files for:
- Bug fixes
- Runtime errors
- Port mismatches
- Hook issues
- Proxy issues
- Temporary debug endpoints

Instead:
- Update docs/architecture/runtime-and-troubleshooting.md
- Or modify existing architecture documentation

Never generate *_FIX.md files.
Never generate *_DEBUG.md files.
Never generate implementation summary files.
